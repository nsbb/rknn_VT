## 테스트 환경

- **하드웨어**: RK3588 (ARM Cortex-A76 × 4 + Cortex-A55 × 4, NPU 6TOPS)
- **OS**: Linux 5.10 (Rockchip BSP)
- **런타임**: RKNN-Toolkit2 v2.3.2 / rknn-toolkit-lite2 v2.3.2 (Python 3.8)
- **모델**: BCResNet-t2-Focal-ep110 (웨이크워드 2-class 분류)
- **테스트 날짜**: 2026-03-12

## 1. NPU 동작 여부

| 항목 | 수정 전 (원본 ONNX → RKNN) | 수정 후 (fixed ONNX → RKNN) |

| RKNN 변환 | 성공 (경고 없음) | 성공 |

## 3. 레이턴시

### NPU 추론 단독 (bench_npu.py, N=300회 평균)

| 설정 | 레이턴시 | 처리량 |
|------|---------|--------|
| ONNX CPU | 2.48 ms | 403/sec |
| NPU Core 0 | 6.21 ms | 161/sec |
| NPU Core 0+1 | 5.59 ms | 179/sec |
| NPU Auto | 6.40 ms | 156/sec |

> NPU 단독 추론은 ONNX CPU보다 느리다. 그러나 NPU는 전용 하드웨어이므로 CPU 부하를 오프로드할 수 있다는 이점이 있다.

### End-to-End (LogMel 전처리 + NPU 추론, bench_e2e.py)

| 항목 | 시간 |
|------|------|
| LogMel 전처리 (numpy) | 13.36 ms |
| NPU 추론 (Core 0) | 6.21 ms |
| **E2E 합계** | **21.30 ms** |
| 처리 속도 | 47 calls/sec |
| Hop 160ms 기준 실시간 여유 | **7.5배** |

슬라이딩 윈도우 hop=160ms 대비 E2E 21ms → hop 예산의 **13%만 사용**. 실시간 처리에 충분한 여유.

## 5. FAR (False Alarm Rate) 측정

> **threshold=0.55 + EMA(α=0.3): FAR = 0.00/hr** (118분 TV 뉴스 기준)

Post-processing 파라미터:
- **Refractory**: 검출 후 2.0초 쿨다운 (연속 오탐 억제)
- **EMA**: 지수이동평균 α=0.3 (순간 피크 억제)
- **N-of-M**: 최근 5프레임 중 3개 이상 양성 (노이즈 억제)
